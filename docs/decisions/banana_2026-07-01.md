# Decisions: Banana or No Banana — Configurable multi-provider player seats
Spec: Planning/specs/banana_spec.md ("Configurable multi-provider player seats" feature)
Date: 2026-07-01

- **Generalized "Box Holder"/"Guesser" into LeftPlayer/RightPlayer seats**, each independently human or AI — user request, explicitly confirmed over the narrower "just make the Box Holder's model configurable" option. This steps into v2 territory (AI-vs-AI) ahead of schedule, but deliberately **not** the rest of v2 (batch runner, deviation-from-50% leaderboard stay deferred) — proportional: build the seat/provider generalization now since it was asked for, leave the analysis tooling for when it's actually wanted.
- **Three providers — ollama / openai_compat / anthropic** — chosen after reviewing the sibling project Polis's LLM abstraction (`H:\Polis\backend\engine\llm\client.py`), which dispatches by a provider string to a single call signature. Same shape here (`llm_providers.chat_stream(messages, cfg, temperature)`), but implemented via raw `httpx` SSE parsing rather than the `anthropic`/`openai` SDKs — Banana's existing `ollama_client.py` already hand-rolls NDJSON streaming in this style, and streaming (not just single-shot `complete()`, which is all Polis does) is core to the Box Holder's "live banter" feel, so the SDKs' typical non-streaming-by-default ergonomics weren't a fit anyway. Avoids adding two new heavyweight dependencies for a local, low-volume game.
- **Secrets in `.env` only, resolved server-side, never sent to the browser** — `GET /api/players` exposes `kind`/`provider`/`model` but never `api_key`. Matches Polis's `.env`/`.env.example` convention (`.env` already gitignored here from the original scaffold). Skipped Polis's DB-backed `LLMProfile` + Fernet-encrypted-key layer — that solves multi-tenant/multi-city key storage, which doesn't apply to a single local player; env vars are proportional here.
- **Per-round overrides stay LeftPlayer-`model`-only** (unchanged from the existing Match settings feature) — `provider`/`base_url`/`api_key`/seat `kind` for both seats are fixed at server-config (`.env`) level. Letting the browser choose provider/credentials at request time would mean either exposing API keys client-side or building a server-side key-selection UI; neither is asked for yet.
- **AI-vs-AI (RightPlayer = AI) implemented as a single non-streamed `POST /api/round/{id}/advance` per turn**, not token-streamed like the human path. The spec already precedents "acceptable to land non-streamed... provided the shipped feature streams" for the *first* commit of an endpoint (§4); autoplay is new-enough territory that non-streamed is the honest first cut. `/say` (the human path) is unchanged and still streams.
- **Forced lock-in with a deterministic fallback** — once turns are exhausted, `/advance` appends an explicit "you must answer now" instruction to the Guesser's messages; if the model still returns something unparseable, the round is defaulted to `NO_BANANA` rather than looping. Rejected an unbounded retry (a misbehaving/cloud-timeout model could hang a round forever) and rejected surfacing a raw error to the player (breaks the game-show frame for no real benefit) — a deterministic default keeps every round provably terminating, which the existing coin-fairness/turn-accounting tests already lean on as an invariant.
- **`prompts/guesser.md`** (already locked, unused until now — SPEC.md called it out for "when the Guesser is AI — v2") is reused as-is for the AI Guesser's system prompt; no changes needed, confirming the original authors anticipated this seat.
