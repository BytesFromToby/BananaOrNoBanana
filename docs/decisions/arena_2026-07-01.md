# Decisions: Community Arena — build adoption & conformance
Spec: Planning/specs/arena_spec.md
Date: 2026-07-01

- **Arena spec adopted for the autonomous build.** The spec was authored ahead of the build
  (commit 0b5d331, amended f6cc111 for standard-conditions). It is the source of truth; the
  build does not re-derive it.
- **Model-alias "Open Question" downgraded to a resolved Assumption.** Per TERMS §6 (Assumption
  vs Open Question: is a wrong default a cheap edit or a rebuild?), grouping strictly by
  provider/model string with no alias map is cheap, low-surprise, and additive-later — so it is
  a resolved v1 default under `## Assumptions`, not a blocking fork. The spec now has zero
  blocking Open Questions.
- **Build executed inline in the main session, not via the Plumbline subagent pipeline.** The
  installed plugin was v1.0.0, whose agent frontmatter used lowercase `tools:` names — invalid,
  so spawned subagents got no tools and no-op'd. Upgraded the plugin to v1.0.1 (which fixes the
  frontmatter), but a plugin update needs a fresh session to load; this session stayed on the
  broken defs. With the user's go-ahead, the pipeline stages (architect conformance → foreman
  blueprint → builder code+tests → inspector proof) were run by hand in the main session. The
  trade, recorded honestly: working, tested code, but the final sign-off is self-assessed rather
  than structurally independent. See output/homeowner/HomeownerLog_2026-07-01_20-26.md.
- **Worker testability without live Cloudflare.** Core Worker logic (validation, server-side
  rescore, aggregation) is factored into pure modules unit-tested with vitest against a fake
  D1/KV, so every `[automated]` item runs offline. Only the `[human-required]` items (live
  deploy, real submission, browser render) need Cloudflare.
